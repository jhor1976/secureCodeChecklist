No incluir claves API o secretos en código fuente; usar secretos gestionados por la nube o CI/CD

Explicación: Es una mala práctica guardar claves API, tokens, contraseñas o secretos directamente en el código fuente (por ejemplo, en .java, .properties o repositorios Git). Esto expone a la organización a fugas de información sensibles, sobre todo cuando el código se comparte o sube a repositorios públicos/privados. En su lugar se deben usar:

  • Vaults o Secret Managers (ej: HashiCorp Vault, AWS Secrets Manager, Google Secret Manager, Azure Key Vault).
  • Inyección segura de variables de entorno desde pipelines de CI/CD (ej: GitLab, GitHub Actions, Jenkins).
  • Archivos de configuración encriptados gestionados fuera del código.

Buenas prácticas:

  • Usar variables de entorno para inyectar secretos.
  • Gestionar secretos con Vaults o Secret Managers de la nube.
  • No versionar archivos con secretos (ejemplo: .env, application.properties).
  • Usar escáneres automáticos como git-secrets o trufflehog.

Ejemplo en Java (Spring WebFlux):


@RestController
public class ApiController {

    @Value("${external.api.key}")
    private String apiKey;

    @GetMapping("/secure-call")
    public String callExternalService() {
        return "Consumir servicio externo con API Key segura (no hardcodeada).";
    }
}

application.yml:


external:
  api:
    key: ${EXTERNAL_API_KEY}

⚡️ En este ejemplo, EXTERNAL_API_KEY se inyecta como variable de entorno en el servidor o CI/CD, nunca en el código fuente.
