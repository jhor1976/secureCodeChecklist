✅ Identificar y validar fuentes de datos

Resumen y objetivo

Esta página contiene: (1) descripción de qué significa clasificar fuentes de datos como confiables o no, (2) criterios para clasificarlas, (3) recomendaciones de validación (4) y buenas practicas.

¿Qué significa?

Fuente confiable: controlada por tu organización y con garantías de integridad y formato (ej.: microservicio interno autenticado, base de datos propia).
Fuente no confiable: cualquier entrada que pueda ser manipulada por terceros o no esté 100% bajo tu control (ej.: formularios, archivos subidos, APIs externas).

Regla práctica

Si no lo controlas al 100%: trátalo como no confiable y aplica validaciones.

Cómo clasificar fuentes (rápido)

  • Entradas directas de usuario: formularios, parámetros URL, cabeceras — Siempre NO confiables.
  • Archivos cargados y dispositivos: CSV, imágenes, IoT — NO confiables.
  • Fuentes internas controladas: DB propia, caches, colas internas — Confiables si el origen está controlado.
  • Servicios autenticados internos: microservicios internos con mTLS/JWT — Confiables, pero validar esquema/contrato.
  • Servicios de terceros: REST/SOAP externos — NO confiables.

Cómo validar datos de fuentes no confiables

  1. Validación sintáctica: tipo, longitud, formato (regex), juego de caracteres (UTF-8).
  2. Validación semántica: reglas de dominio (ej.: edad ≥ 18, montos > 0, fechas coherentes).
  3. Validación de seguridad: proteger contra inyección, XSS, path traversal, bytes nulos, doble codificación.
  4. Listas permitidas (whitelists): preferir whitelist sobre blacklist.
  5. Canonicalización: normalizar antes de validar para evitar doble codificación y ofuscación.
  6. Centralización: reunir reglas en una librería o servicio común (evita duplicidad y errores).
  7. Logging y alertas: registrar rechazos, patrones anómalos y origen de los datos.

Buenas prácticas

  • Centraliza validaciones en una librería o middleware (filtros/interceptors) para asegurar consistencia.
  • Registra los rechazos con suficiente contexto para investigar sin exponer datos sensibles.
  • Define quién es responsable de mantener las listas permitidas y las reglas de validación (propiedad del equipo/producto).
  • Incluye estas pruebas en tu pipeline de CI/CD y crea tests automatizados que cubran inputs válidos e inválidos.
